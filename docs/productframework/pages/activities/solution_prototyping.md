## Description

Проектирование конкретных решений начинается с проектирования опыта взаимодействия с потребителем. Рализация этого опыта может отличаться в разных продуктах, тем самым с разной эффективностью влияя на удовлетворение потребностей.

## Extracted Text

SOLUTION PROTOTYPING

Какими свойствами должен обладать продукт, чтобы приносить ценность потребителю?

Артефакты и документы

База со статическими и динамическими прототипами, предназначенными для проектирования интерфейсов взаимодействия с пользователем и валидации гипотез отдельных решений внутри продукта.

Описание профиля пользователя

Характеристики и поведение тех, кто непосредственно пользуется продуктом. Используется для оптимизации user и customer experience.

Модели и инструменты

Customer Journey Map

Модель описывает путь клиента, который он проходит при взаимодействии с компанией, во всех точках контакта. Модель применяется для выявления проблем клиента на основании его целей, опыта и ожиданий в каждой точке взаимодействия. Часто используемые разрезы: описание точки контакта, цели клиента, метрики этапа, процессы компании, действия клиента, эмоции, что он чувствует и думает, проблемы клиента, ожидания клиента, наблюдения.

Модель описывает поток взаимодействия клиента с сервисом, фокусируясь на обеспечивающих это взаимодействие процессах. Модель применяется для выявления угроз сервису и определения возможностей оптимизации. Разрезы: действия клиентов до взаимодействия с сервисом и во время взаимодействия с ним, действия сотрудников, обеспечивающих взаимодействие с клиентом, но остающиеся для него скрытыми, а также процессы компании.

Модель последовательности взаимодействия пользователя с интерфейсом продукта для достижения своей цели. Каждый из шагов user flow обусловлен мотивацией пользователя, которую определяет потребности потребителя и верхнеуровневый value stream. User flow определяет, каким конкретно образом реализованы сценарии использования продукта.

Карта экранов отображает реализацию user flow продукта при помощи конкретных интерфейсных решений. Карта включает в себя перечень состояний каждого элемента интерфейса и переходы между ними, инициированные пользователем или системой. Хорошая карта экранов собрана при помощи дизайн-системы компании и используется как для постановки требований в разработку, так и в качестве артефакта документации.

Процесс проектирования решений в виде модели для проверки гипотез взаимодействия при помощи обратной связи от пользователей. Цель прототипирования - сокращение рисков использования продукта уже на этапе анализа до фактического релиза. Хороший прототип должен быть дешевым, быстро создаваемым и включать только то, что необходимо для валидации гипотез. Как правило, прототипы создаются для программных интерфейсов, но могут быть использованы вне зависимости от вида точки взаимодействия с пользователями.

Шаблон для описания процесса работы с функционалом продукта. Основной упор - на этапе проектирования и внедрения фичи в продукт. Включает в себя блоки: описание идей, пять почему, проблемы пользователей, контекстные ситуации, ценность, текущие возможности, текущие ограничения, кого затрагивает внедрение фичи, их текущие привычки, задачи пользователей, осведомленность, помощь, цели бизнеса, критерии достижения успеха (метрики), ключевые активности.

Автор модели: Никита Ефимов

Формат описания потребностей клиентов, в котором основной акцент стоит на контексте, при котором возникают эти потребности. Формат был предложен как альтернатива user story и смещает фокус с особенностей портрета клиента на контекст работы, которую ему нужно выполнить, и почему эта работа для него важна.
